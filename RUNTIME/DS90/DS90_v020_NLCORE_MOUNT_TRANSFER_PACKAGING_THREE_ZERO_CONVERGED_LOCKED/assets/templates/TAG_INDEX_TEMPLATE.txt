# TAG_INDEX_TEMPLATE.txt
# 対象小説プロジェクト用 TAG_INDEX.txt テンプレート

区分: 横断検索索引テンプレート
役割: 検索タグから棚TXT内の設計項目へ戻るための索引
注意: このテンプレート自体は作品正本ではない。
注意: 実際の TAG_INDEX.txt は対象小説プロジェクト側へ置く。
注意: 090_DSカプセルへ作品情報として保存しない。

---

## 使い方

マウント移管時に、反映した設計項目を1項目1ブロックで登録する。

棚TXT内の該当項目にも、近接して `検索ID:`、`タグ棚:`、`関連タグ棚:`、`検索タグ:`、`検索索引:` を置く。

## タグ棚の基本分類

- 世界軸: #WORLD:
- 背景: #BG:
- 人物: #CHAR:
- 関係: #REL:
- 組織: #ORG:
- 地理: #GEO: / #PLACE:
- 距離・物流: #DIST: / #ROUTE: / #LOGISTICS:
- 時間・章幕: #TIME: / #ARC: / #EP:
- 筋・火種: #THREAD: / #SEED:
- 生活・具体物: #LIFE: / #ITEM: / #OP:
- 文体・話レイヤー: #STYLE: / #LAYER:
- 禁止線: #BAN:
- 作業LOG: #LOG:

`#DS:` は設計項目の種類を示す管理寄りタグであり、世界軸、背景、人物、組織、地理、距離を同じ棚に押し込むために使わない。

## まとめ検索の意味

TAG_INDEX.txtは、タグ一致だけでなく関係逆引きに使う。

組織名で検索したら、その組織の定義だけでなく所属人物、関係人物、拠点、管轄、未確認関係を引けるようにする。

場所名で検索したら、その場所の定義だけでなく、そこにいる人物、そこにいる組織、距離・物流、生活圏を引けるようにする。

人物名で検索したら、人物定義だけでなく、所属、配置、関係、疑惑、未確認を引けるようにする。

「組織のここには此奴がいる」まで引けない索引は未完成として扱う。
---

## 索引補修候補

検索時またはマウント移管時に、引けない関係が見つかった場合は、次の形で補修候補を置く。

[INDEX_REPAIR]
補修ID:
対象:
不足種別: ITEM / ENTITY / RELATION / タグ棚 / 検索タグ / 所在
理由:
補修案:
根拠:
状態: 未確認 / 要反映 / 要確認
最終確認:

例。

[INDEX_REPAIR]
補修ID: REPAIR-YYYYMMDD-001
対象: <組織名>
不足種別: RELATION
理由: 組織定義はあるが所属人物が逆引きできない。
補修案: <人物名>との所属RELATIONを追加候補にする。
根拠: 棚TXT / ユーザー発話 / 未確認
状態: 要確認
最終確認: 2026-05-25

## GPT側協力

TAG_INDEXはユーザーが手で全部作る前提にしない。

設計さんは、棚反映時にITEM/ENTITY/RELATIONを可能な範囲で組み立てる。

ユーザーが自然文で検索した場合も、タグ名へ変換して引く。

引けない場合は、索引不足と補修候補を返す。


---

## AI使用者前提の追加欄

TAG_INDEXは、ユーザーが手で読むためだけの一覧ではない。

主な使用者は、設計さん、執筆さん、通し修正番、次個体である。

後続AIが自然文から引けるように、必要に応じて次の欄を使う。

```txt
別名:
検索ヒント:
逆引き優先:
AIメモ:
索引補修候補:
```

使い分け。

- `別名:` 正式名、仮名、呼称、旧名、ユーザー呼び、作業名。
- `検索ヒント:` 次個体やユーザーが言いそうな自然文。
- `逆引き優先:` 組織→人物、場所→人物、人物→所属など、先に辿るべき方向。
- `AIメモ:` 検索補助だけに使う短い注意。
- `索引補修候補:` 引けない関係、未登録ENTITY、未登録RELATION。

`AIメモ:` は正本ではない。

`AIメモ:` に長文裁定を蓄積しない。

未確認の別名や関係は、状態を未確認にして扱う。


## 逆引き優先例

```txt
逆引き優先:
- 組織→所属人物
- 組織→拠点
- 組織→交渉窓口
- 場所→所在人物
- 場所→所在組織
- 人物→所属組織
- 人物→配置場所
- 人物→関係人物
- 距離→場所
- 背景→関係人物
```

## AI省力精度欄

後続AIが少ない労力で100%に近づくため、必要に応じて次を使う。

```txt
AI優先入口:
自然文検索例:
欠損時の次手:
```

使い方。

- `AI優先入口:` 後続AIが最初に読むべきITEM/ENTITY/RELATION。
- `自然文検索例:` ユーザーや次個体が言いそうな聞き方。
- `欠損時の次手:` 引けない時に何を補修するか。

これらは検索補助であり、正本根拠ではない。

---

## AI使用性検収メモ

TAG_INDEX作成時は、次が引けるか確認する。

```txt
人物 -> 所属 / 配置 / 関係 / 未確認
組織 -> 所属人物 / 拠点 / 窓口 / 未確認関係
場所 -> 所在人物 / 所在組織 / 距離・物流
背景 -> 関係人物 / 関係組織 / 筋・火種
禁止線 -> 効く人物 / 組織 / 場所 / 筋
```

引けない場合は `[INDEX_REPAIR]` に補修候補を置く。

---

## STOP_TAGエントリ形式

STOP_TAGは、禁止線を本文へ長文展開せず、作業対象の索引や本文近接で踏ませるためのタグである。

STOP_TAGの意味本文は `STOP_TAG_INDEX.md` へ置く。

TAG_INDEX側には、対象項目に関係するSTOP_TAGだけを書く。

```txt
STOP_TAG: #stop:<短い安定キー>
STOP_TAG_INDEX: STOP_TAG_INDEX.md / #stop:<短い安定キー>
```

STOP_TAGは正本根拠ではない。

STOP_TAGを本文条件の代替にしない。

未定義STOP_TAGを使わない。
