## 0. 位置づけ

このファイルは、検索タグを「付ける」だけで終わらせず、後から引ける状態にするための規格である。

検索タグは検索導線である。

検索タグは正本根拠ではない。

検索タグは本文条件源ではない。

検索タグで未確認を確定しない。

タグは、棚TXT内の設計項目近接に置く。

索引は、タグ付き設計項目を横断検索するために置く。

---

## 1. 必須原則

すべての設計項目は、次の2つを持つ。

1. `検索タグ:`
2. `検索索引:` または TAG_INDEX.txt 内の索引行

タグだけでは完了扱いしない。

タグがあっても、引けない場合は検索導線未完成である。

設計項目を棚へ反映する場合、その項目を TAG_INDEX.txt から引ける形にする。

## 1.6 タグ棚分類

検索タグは1枚の平たいタグ欄では扱わない。

世界軸、背景、人物、組織、地理、距離は同じ棚のタグではない。

設計項目には `タグ棚:` を置く。

`タグ棚:` は、その項目を最初に引く入口である。

`検索タグ:` は、その項目を横断で拾うための札である。

`#DS:` は設計項目の種類であり、内容分類の全部を背負わせない。

タグ棚の基本分類。

```txt
タグ棚: 世界軸
用途: 世界構造、外圧、時代圧、文明差、宗教圧、制度圧。
主タグ: #WORLD:

タグ棚: 背景
用途: 過去事情、傭兵期間、事件前史、隠れている前提、明かしすぎない事情。
主タグ: #BG:

タグ棚: 人物
用途: キャラ核、弱点、反応、関係変化、個人の未確認。
主タグ: #CHAR:

タグ棚: 関係
用途: 二者関係、距離変化、上下、依存、警戒、買収疑惑、対立。
主タグ: #REL:

タグ棚: 組織
用途: 所属、派閥、宗教組織、外圧主体、内部序列、交渉窓口。
主タグ: #ORG:

タグ棚: 地理
用途: 場所、街区、地域、窓、拠点、移動先、地形。
主タグ: #GEO: / #PLACE:

タグ棚: 距離・物流
用途: 移動距離、到達時間、補給線、交通、連絡可能性、生活圏。
主タグ: #DIST: / #ROUTE: / #LOGISTICS:

タグ棚: 時間・章幕
用途: 年代、章、幕、帯、話数、日次、順序。
主タグ: #TIME: / #ARC: / #EP:

タグ棚: 筋・火種
用途: 話の筋、伏線、火種、残響、次話導線。
主タグ: #THREAD: / #SEED:

タグ棚: 生活・具体物
用途: 小物、手順、台所、仕事、生活着地、置き場所。
主タグ: #LIFE: / #ITEM: / #OP:

タグ棚: 文体・話レイヤー
用途: 地の文、温度、観測、話者札、説明量、禁止群。
主タグ: #STYLE: / #LAYER:

タグ棚: 禁止線
用途: やらない展開、NG処理、便利化禁止、感情代替禁止。
主タグ: #BAN:

タグ棚: 作業LOG
用途: 熱源LOG、条件化LOG、シミュレーションLOG、移管LOG。
主タグ: #LOG:
```

1項目が複数棚にまたがる場合は、主棚と関連棚を分ける。

```txt
タグ棚: 人物
関連タグ棚: 組織 / 背景
```

主棚は、その項目を最初に探す場所である。

関連タグ棚は、横断検索で拾う補助導線である。

タグ棚は正本根拠ではない。

タグ棚で未確認を確定しない。

タグ棚が未記載の設計項目は、検索設計未完了として扱う。


## 2. 検索ID

設計項目には、必要に応じて検索IDを付ける。

基本形。

```txt
検索ID: DS-<棚番号>-<YYYYMMDD>-<連番>
```

例。

```txt
検索ID: DS-094-YYYYMMDD-001
検索ID: DS-092-YYYYMMDD-014
検索ID: DS-028-YYYYMMDD-003
```

検索IDは、同一項目を後から引くための短い固定札である。

検索IDは正本根拠ではない。

検索IDを理由に内容を確定しない。

## 3. TAG_INDEX.txt

対象小説プロジェクトのマウントZIPには、検索索引用の `TAG_INDEX.txt` を作成または更新する。

`TAG_INDEX.txt` は作品棚の横断索引である。

`TAG_INDEX.txt` は作品本文ではない。

`TAG_INDEX.txt` は正本の代替ではない。

`TAG_INDEX.txt` は棚TXT内の項目へ戻るための導線である。

`TAG_INDEX.txt` は090_DSカプセル内へ保存しない。

`TAG_INDEX.txt` は対象小説プロジェクト側へ置く。

---

## 4. TAG_INDEX.txt エントリ形式

1項目1ブロックで記録する。

```txt
[ITEM]
検索ID:
表示名:
所在: <棚ファイルパス>#<見出しまたはID>
棚:
状態:
検索タグ:
短要約:
根拠:
関連ID:
最終反映:
```

最低限必要な項目。

```txt
検索ID:
所在:
棚:
状態:
タグ棚:
検索タグ:
短要約:
```

`所在` は、次個体が該当棚TXTへ戻れる形にする。

`短要約` は検索結果一覧用であり、本文条件の代替ではない。

## 5. 棚TXT側の近接表記

棚TXT内の設計項目は、原則として項目直下に次を置く。

```txt
検索ID:
タグ棚:
関連タグ棚:
検索タグ:
検索索引: TAG_INDEX.txt / <検索ID>
```

例。

```txt
### カイの買収疑惑
検索ID: DS-028-YYYYMMDD-002
タグ棚: 人物
関連タグ棚: 関係 / 組織
検索タグ: #DS:未確認 #SHELF:028 #STATE:未確認 #CHAR:カイ #REL:買収疑惑 #ORG:未確認 #SOURCE:ユーザー発話
検索索引: TAG_INDEX.txt / DS-028-YYYYMMDD-002
```

## 5.5 STOP_TAG規格

STOP_TAGは、禁止線を長文リスト化せず、該当する索引や本文近接に付けるためのタグである。

STOP_TAGは検索タグとは別に扱う。

STOP_TAGは正本根拠ではない。

STOP_TAGは、作業時に踏むべき危険線の導線である。

STOP_TAGの意味本文は、対象プロジェクト側の `STOP_TAG_INDEX.md` へ置く。

090_DS内に作品固有STOP_TAG辞書を蓄積しない。

### STOP_TAGの表記

基本形。

```txt
STOP_TAG: #stop:<短い安定キー>
STOP_TAG_INDEX: STOP_TAG_INDEX.md / #stop:<短い安定キー>
```

例。

```txt
STOP_TAG: #stop:no_chapter_drop #stop:no_auto_mount
STOP_TAG_INDEX: STOP_TAG_INDEX.md / #stop:no_chapter_drop
```

### STOP_TAGの原則

- 未定義STOP_TAGを使わない。
- 同義STOP_TAGを増やさない。
- STOP_TAGを本文条件の代替にしない。
- STOP_TAGだけで設計判断しない。
- 重大な全域禁止はGLOBAL_STOPにも短文で残す。
- 作品固有STOPは、対象作品の正本インデックスまたは棚インデックスへ付ける。
- 本文へ直行せず、対象インデックスのSTOP_TAGを確認してから本文へ進む。

### STOP_TAG_INDEXの最小形式

```txt
[STOP_TAG]
タグ:
表示名:
意味:
適用範囲:
付与先:
関連正本インデックス:
解除条件:
同義禁止:
備考:
```

解除条件が未記載の場合、設計さんはそのSTOP_TAGを解除済み扱いしない。


## 6. タグ検索要求への対応

ユーザーがタグ、人物名、場所名、筋名、話ID、棚番号、検索IDで探した場合、設計さんは検索応答に入る。

検索要求の例。

```txt
#CHAR:カイ を引いて
タグ棚:人物 の未確認を出して
タグ棚:組織 の外圧関連を見せて
#DIST:徒歩圏外 を見せて
#REL:買収疑惑 を見せて
DS-028-YYYYMMDD-002 はどこ？
カイ関連の未確認を出して
024の起点タグを一覧にして
```

対応手順。

1. TAG_INDEX.txt がある場合は、まずTAG_INDEX.txtを検索する。
2. 一致候補を一覧化する。
3. 必要に応じて所在の棚TXTを実読する。
4. TAG_INDEX.txt がない場合は、棚TXT内の `検索タグ:` を直接検索する。
5. 棚TXTにタグがあるのに索引がない場合は、`索引未登録` として補正候補を出す。
6. 見つからない場合は、タグなし項目の可能性を明示する。

## 7. 検索結果の返し方

検索結果は短く返す。

```txt
検索結果:
- 検索ID:
  所在:
  状態:
  タグ棚:
  関連タグ棚:
  タグ:
  短要約:
  次に読む:
```

複数ある場合は、状態順で並べる。

優先順。

1. 採用
2. 反映済み
3. 保留
4. 未確認
5. 仮置き
6. 推測
7. 後送
8. 捨てる

検索結果だけで内容を確定しない。

内容判断が必要な場合は、所在の棚TXTを読む。

## 7.5 まとめ検索 / 関係逆引き

検索は、タグ一致だけで終わらせない。

ユーザーが人物、組織、場所、距離、背景、世界軸、章幕、話ID、検索ID、自然文で探した場合、設計さんは `まとめ検索` として扱う。

まとめ検索では、次を同時に見る。

1. TAG_INDEX.txt の `[ITEM]`
2. TAG_INDEX.txt の `[ENTITY]`
3. TAG_INDEX.txt の `[RELATION]`
4. 棚TXT内の `検索タグ:`
5. 棚TXT内の `関係索引:`

検索対象が組織の場合、最低限次を引く。

- その組織の定義
- 所属人物
- 関係人物
- 拠点
- 管轄地
- 対立組織
- 交渉窓口
- 未確認の所属
- 後送/保留の関係

検索対象が場所の場合、最低限次を引く。

- その場所の定義
- そこにいる人物
- そこにいる組織
- 距離・物流
- 生活圏
- 進入禁止/移動制限
- 未確認の配置

検索対象が人物の場合、最低限次を引く。

- 人物定義
- 所属組織
- 配置場所
- 関係人物
- 関係組織
- 買収/疑惑/敵対/保護/協力
- 未確認

`組織のここには此奴がいる` まで引けない検索は、検索設計未完成として扱う。

## 7.6 実体索引

TAG_INDEX.txt には、必要に応じて `[ENTITY]` を置く。

`[ENTITY]` は人物、組織、場所、世界軸、背景、制度、筋などの実体を引くための索引である。

基本形。

```txt
[ENTITY]
実体ID:
実体種別:
表示名:
別名:
所在:
状態:
タグ棚:
検索タグ:
短要約:
関連ITEM:
関連RELATION:
最終反映:
```

実体IDの基本形。

```txt
ENT-<種別>-<YYYYMMDD>-<連番>
```

例。

```txt
ENT-ORG-YYYYMMDD-001
ENT-CHAR-YYYYMMDD-004
ENT-PLACE-YYYYMMDD-002
```

`[ENTITY]` は正本本文の代替ではない。

`[ENTITY]` は棚TXTへ戻るための入口である。

## 7.7 関係索引

TAG_INDEX.txt には、必要に応じて `[RELATION]` を置く。

`[RELATION]` は人物、組織、場所、背景、距離、物流などの間を逆引きするための索引である。

基本形。

```txt
[RELATION]
関係ID:
関係種別:
主体ID:
主体名:
主体タグ棚:
対象ID:
対象名:
対象タグ棚:
状態:
所在:
検索タグ:
短要約:
根拠:
関連ID:
最終反映:
```

関係IDの基本形。

```txt
REL-<YYYYMMDD>-<連番>
```

関係種別の例。

```txt
所属
配置
所在
拠点
管轄
距離
物流
連絡
敵対
協力
保護
買収疑惑
交渉窓口
宗教接触
外圧接触
前史
禁止線
```

例。

```txt
[RELATION]
関係ID: REL-YYYYMMDD-001
関係種別: 所属
主体ID: ENT-CHAR-YYYYMMDD-004
主体名: カイ
主体タグ棚: 人物
対象ID: ENT-ORG-YYYYMMDD-001
対象名: <組織名>
対象タグ棚: 組織
状態: 未確認
所在: 028_H_vXXX/XX_PENDING.txt#カイ所属未確認
検索タグ: #RELTYPE:所属 #CHAR:カイ #ORG:<組織名> #STATE:未確認 #SOURCE:ユーザー発話
短要約: カイが当該組織に属するかは未確認。
根拠: ユーザー発話
関連ID:
最終反映: 2026-05-25
```

関係索引は双方向に検索できるように書く。

人物から組織を引ける。

組織から人物を引ける。

場所から人物を引ける。

人物から場所を引ける。

## 8. マウント移管時の索引更新

マウント移管では、反映した設計項目を TAG_INDEX.txt に登録する。

既存TAG_INDEX.txtがある場合は更新する。

既存項目の状態が変わった場合は、同じ検索IDを維持して状態だけ更新する。

同一項目を重複登録しない。

削除候補や捨てる項目も、必要なら `#STATE:捨てる` として引けるようにする。

未確認は `#STATE:未確認` のまま引けるようにする。

---

## 9. 棚掃除時の索引扱い

棚掃除ではZIP生成しない。

棚掃除ではTAG_INDEX.txtを実ファイル更新しない。

ただし、プロジェクト側再開メモまたはEND_LOG未反映メモから棚TXTまたはLOGファイルへ逃がす候補には、検索タグと検索ID候補を付ける。END_LOG自体を作品条件源にはしない。

棚掃除中の索引候補は、マウント移管時にTAG_INDEX.txtへ反映する。

## 10. STOP

STOP:
- 検索タグだけで「引ける形」扱いしない。
- TAG_INDEX.txtがあるのに読まずに検索完了扱いしない。
- タグがない設計項目を完成扱いしない。
- 索引未登録の項目を反映完了扱いしない。
- TAG_INDEX.txtを090_DSカプセルへ作品正本として埋め込まない。
- 検索結果だけで未確認を確定しない。

---

## 11. 完了条件

設計項目の検索対応完了条件。

- 棚TXT内の項目近接に `タグ棚:` と `検索タグ:` がある。
- 必要に応じて `検索ID:` がある。
- TAG_INDEX.txtからタグ棚、検索ID、検索タグで該当項目を引ける。
- TAG_INDEX.txtの所在から棚TXTへ戻れる。
- 状態タグが未確認/採用/保留/捨てる等を維持している。

## 14. まとめ検索の応答最低形

まとめ検索では、タグ一致だけで終わらない。

最低限、次の形で返す。

```txt
検索対象:
-

引けた実体:
-

引けた関係:
-

所属 / 配置 / 所在:
-

関連する未確認:
-

戻る棚:
-

索引不足:
-

補修候補:
-
```

該当なしの場合も、単に「ありません」で終わらない。

TAG_INDEX未整備、タグ不足、RELATION不足、棚未読のどれが原因かを分ける。

---

## 15. 索引補修義務

検索時に次を見つけた場合、設計さんは補修候補を出す。

- タグはあるがTAG_INDEXにない。
- ITEMはあるがENTITYがない。
- ENTITYはあるがRELATIONがない。
- 組織はあるが所属人物が引けない。
- 場所はあるが所在人物/組織が引けない。
- 人物はあるが所属/配置/関係が引けない。
- 関係が自然文にあるのに検索タグへ落ちていない。

補修候補は、勝手に確定しない。

マウント移管時に反映できる形で、`索引補修候補` として出す。

---

## 16. 厳密完了条件

検索対応完了は、次を満たす状態である。

```txt
- 棚TXT内の項目に タグ棚 / 検索タグ / 検索索引 がある。
- TAG_INDEX.txtに [ITEM] がある。
- 人物、組織、場所、世界軸などの実体に [ENTITY] がある。
- 人物↔組織、人物↔場所、組織↔場所、距離↔場所などの関係に [RELATION] がある。
- まとめ検索で、定義だけでなく関係先まで引ける。
- 索引不足がある場合、補修候補を返せる。
```

タグを付けただけでは未完了である。

検索できそうで引けない状態は未完了である。

## 17. AI使用者前提の検索設計

この検索仕様の主な使用者はユーザーではなく、設計さん、執筆さん、通し修正番、次個体である。

ユーザーにタグ体系を覚えさせるための仕様ではない。

AIが後から楽に、速く、漏れなく引くための仕様である。

検索導線は、ユーザー発話の表記そのものより、後続AIが自然に探す単位を優先して組む。

目的は、楽をして精度を落とすことではない。

目的は、楽に100%へ近づくことである。

### 17.1 AIが楽になる索引の優先順位

索引作成時の優先順位。

1. 後続AIが自然文で探しそうな入口を作る。
2. 人物、組織、場所、距離、背景、世界軸を相互に辿れるようにする。
3. 正式名、仮名、呼称、表記ゆれを別名として拾えるようにする。
4. 未確認、保留、仮置き、推測を状態付きで引けるようにする。
5. 検索結果から棚TXTへ戻れるようにする。
6. 引けない関係は索引補修候補として残す。

ユーザーが指定したタグだけで索引を作らない。

設計さんは、後続AIが検索しやすい別名、自然文検索ヒント、逆引き関係を補助してよい。

ただし、補助は検索導線であり、正本化ではない。

### 17.2 自然文検索ヒント

TAG_INDEX.txt には、必要に応じて `検索ヒント:` を置く。

検索ヒントは、ユーザーや次個体が言いそうな自然文を短く置く。

例。

```txt
検索ヒント:
- カイはどこ所属？
- この組織に誰がいる？
- 外圧の窓口は誰？
- 宗教接触している人物は？
- 徒歩圏外の拠点は？
```

検索ヒントは、表記ゆれ吸収のための導線である。

検索ヒントは正本根拠ではない。

### 17.3 別名 / 呼称 / 仮名

人物、組織、場所、制度には、必要に応じて `別名:` を置く。

```txt
別名: 呼称 / 仮名 / 旧名 / ユーザー呼び / 作業名
```

正式名が未確認の場合でも、作業上の呼び名を `状態: 未確認` で登録してよい。

未確認名を正式名扱いしない。

別名は検索導線であり、確定名ではない。

### 17.4 AI側の先回り登録

設計さんは、次の関係を見つけたらユーザー指示を待たず、索引候補を作る。

- 人物と組織が同じ文脈に出た。
- 人物と場所が同じ文脈に出た。
- 組織と場所が同じ文脈に出た。
- 人物と背景事件が同じ文脈に出た。
- 距離、移動、物流、接触可能性が出た。
- 禁止線が人物、組織、場所、話に関係している。

確定根拠がない場合は `状態: 未確認` または `状態: 仮置き` で候補化する。

候補化しないまま忘れることを避ける。

### 17.5 検索の手抜き禁止

AIが検索しやすくするための手間は省かない。

以下を手抜き扱いする。

- タグだけ付けて逆引きを作らない。
- 組織定義だけ作って所属人物を引けない。
- 場所定義だけ作って所在人物を引けない。
- 人物定義だけ作って所属/配置/関係を引けない。
- 別名や仮名が多いのに別名登録しない。
- 自然文で聞かれた時にタグ指定をユーザーへ要求する。
- 引けない理由を索引補修候補へしない。

楽にするための仕様を作ったなら、AI側が使う。

使わない索引は未完成である。

### 17.6 100%近似の検索手順

検索時は、1回のタグ一致で終えない。

最低限、次の順で広げる。

1. 完全一致。
2. 別名一致。
3. タグ棚一致。
4. ENTITY一致。
5. RELATION一致。
6. 検索ヒント一致。
7. 関連棚の所在を読む。
8. 索引不足を補修候補にする。

該当なしの場合も、次を分けて返す。

```txt
該当なし:
- 本当に項目がない可能性
- タグ未登録の可能性
- ENTITY未登録の可能性
- RELATION未登録の可能性
- 棚未読の可能性
- 索引補修候補
```

「ありません」で終わらない。

### 17.7 AI用最小追加欄

TAG_INDEX.txt の各ブロックには、必要に応じて次を追加してよい。

```txt
別名:
検索ヒント:
逆引き優先:
AIメモ:
索引補修候補:
```

`AIメモ:` は検索補助だけに使う。

`AIメモ:` に作品正本や長文裁定を蓄積しない。
