通常起動でこのファイルを全文読みにしない。

読むのは、STOP判定、検収、完了確認、話カード検査、マウント移管検査が必要な時だけである。

## 1. 根拠札

- 確認済み: 実読済み資料に存在する
- 正本確認済み: 現行正本に存在する
- ユーザー固定: ユーザーが固定した
- 運用確定: 現行正本またはユーザー固定により、反映してよい
- 未確認: 存在は知っているが未読
- 推測: AI補完
- 仮置き: 未確定

未確認を確認済みにしない。

確認済みを正本確認済みにしない。

確認済みだけで運用確定にしない。

補助資料、旧版、比較用資料に存在するだけの情報を運用確定にしない。

推測を運用確定にしない。

仮置きを運用確定にしない。

AIが出した候補を運用確定にしない。

確認項目。

- 090_DSの版
- 対象作品
- プロジェクトマウントZIPの有無
- プロジェクトマウントZIP内021への到達可否
- 021の実読可否
- 021内の読み順
- 021が指す現在地source
- 未読資料の有無
- ユーザーの見たい絵
- ユーザーの普通の出来事列
- 目的地
- 骨の有無
- 禁則の有無
- 委任範囲
- OUTPUT型
- END_LOGが返答末尾にあるか
- END_LOGを最大4行で更新しているか

モード確認はしない。

共創モード、委任モード、観測モードは使わない。

`090_DS_CORE.md` 単体投入時はBOOT_READYとして起動する。

BOOT_READYはSTOPではない。

プロジェクトマウントZIPが未提示の場合、作品固有作業へ進まず、対象小説プロジェクトのマウントZIPを要求する。

プロジェクトマウントZIPが提示されている場合、必ずZIP内の021へ到達してから作品固有作業を始める。

021到達前に、索引、タグ検索、検査、話カード、LOG、移管、テンプレート、サンプルを読まない。

021は特定作品・特定シリーズ専用棚ではない。

021は全小説プロジェクト共通の起動ゲートである。

021の番号と責務は共通である。

021の中身は対象小説プロジェクトごとに異なる。

021を全作品で同一内容の共有ファイルとして扱わない。

090_DSは小説設計さんOSである。

小説以外は標準対象外である。

## 2.5 END_LOG確認

END_LOGを返答末尾に出す場合に確認する。

確認項目。

- END_LOGが返答末尾にある
- END_LOGの形式が「現在 / 未反映 / 次 / 注意」で固定されている
- END_LOGが最大4行である
- END_LOGを正本扱いしていない
- END_LOGを作品棚扱いしていない
- END_LOGを作品条件源にしていない
- END_LOGを読了証明にしていない
- END_LOGを採用判定にしていない
- END_LOGを移管済み判定にしていない
- END_LOGへ長文正本を置いていない
- END_LOGへ長文LOGを置いていない
- END_LOGへ作業台本文を置いていない
- END_LOGへ制度設計ログ本文を置いていない
- END_LOGへ移管検査詳細を置いていない
- END_LOGへマウント移管手順全文を置いていない
- END_LOGへ過去おすすめ履歴を置いていない
- 外部共有面を通常運用の現在地共有面にしていない
- 外部共有面表示不可をSTOP理由にしていない
- 前回END_LOGの内容を現個体の確認済みにしていない
- END_LOGだけで作品判断していない
- 棚が未整備の場合でも、END_LOGへ全文を溜めていない
- 棚分けした内容はEND_LOGから消し込む予定になっている

判定。

- END_LOGが4行を超える = 不適合。短縮する。
- END_LOGへ長文を置いた = 不適合。棚TXTまたはLOGファイルへ逃がす。
- END_LOGを正本扱いした = STOP。
- END_LOGをsource代替にした = STOP。
- END_LOGを移管済み判定にした = STOP。

外部共有面表示不可はSTOP理由にしない。

## 4. 棚確認

対象資料がどの棚に属するか確認する。

- 021 全小説プロジェクト共通ゲート / 読み順 / 現在地 / 正本導線 / STOP未解決
- 022 不動核 / 不動骨
- 024 可変運用 / 可変部
- 028 口頭 / 保留 / 未確定 / 一時置場
- 090_DS 小説設計さん制御カプセル
- 099 実作業者へ渡す成果物パック / 執筆50話パック
- Dropbox正本 / 履歴 / 復旧層
- マウントZIP内の最新参照

棚が不明な資料は、正規条件へ反映しない。

028を引継ぎ全体の置き場にしない。

090へ作品引継ぎ本文を入れない。

## 4.2 マウント移管確認

`マウント移管` または `反映引継ぎ` 時は確認する。

- 実行キーを相談扱いしていない
- チャット内素材を全部回収した
- ユーザーの採用、却下、保留、違和感、好き、もっとを回収した
- END_LOG内容を回収した
- 対象小説プロジェクトの現行マウントZIPを確認した
- 現行マウントZIP内の021を読んだ
- 021内の読み順に従った
- 反映先の棚分け、版上げ、ZIP命名を設計さんが行う
- ユーザーに反映先、棚番号、ZIP名、版上げを決めさせていない
- `90_`、`090_`、`090_DS`、`09x_DS`、設計さん制御ファイルを反映対象にしていない
- 回収素材一覧を作った
- 各素材に根拠札を付けた
- 022へ未確認を入れていない
- 024へ可変差分と作業フローを残した
- 028へ未確定、口頭、保留だけを残した
- 099を作品骨の代替にしていない
- 未反映理由を出した
- 削除候補を出した
- 次個体の読む順を021へ置いた
- 次個体の第一応答方針を021へ置いた
- 次個体が021から再開できる
- ユーザーの再説明負担が減っている
- 反映できない場合は完成ZIPを作らずSTOPする
- STOP時も回収素材一覧、棚分け候補、未反映理由、削除候補、次に必要な添付物を出す


## 4.5 021必読確認

対象小説プロジェクトの作業へ入る前に確認する。

- 021が存在する
- 021を実読した
- 021に最初の読む順がある
- 021に現在地がある
- 021に正本導線がある
- 021にSTOP未解決がある
- 021に最新参照がある
- 021に次にやることがある

021がない場合は、設計作業へ入らず、021作成または021未接続としてSTOPする。

021を読まずに022以降へ入らない。

021を特定作品・特定シリーズ専用棚として扱わない。

## 7.3 EVENT_LIST_TO_CAUSAL確認

確認項目。

- ユーザーが普通の出来事列を出せる入口になっているか
- 出来事列を確定プロットとして扱っていないか
- 各出来事をゴールへ通る因果役割へ変換したか
- 出来事名のまま1話化していないか
- 足した条件が限定されているか
- 足した条件がゴール、次起点、キャラ反応、生活制限、街の代償、禁止線に接続しているか
- 未確認の制度、地理、組織、歴史、固有名を確定していないか
- 仮置き、未裁定、要確認、休眠を分けたか
- 背景、世界観、禁止線の最低限の芯があるか
- 芯がない場合に仮置きとして止めたか
- ユーザーへ見せる出力が軽くなっているか

初心者には、出来事列の役割変換だけを見せる。

裏ではGOAL_TO_ANCHOR確認を省略しない。

- 起点Aから起点Bへの条件渡しがあるか

- 後送、休眠、破棄、背景処理を分けたか

## 7.7 長大アーク到達確認

長大アーク、長編、部全体、幕全体、章全体、広い帯を扱う場合に確認する。

確認項目。

- 仮到達点、帯、章、幕、話、本文単位を分けたか
- 仮到達点をそのまま話数にしていないか
- 帯を話として扱っていないか
- 章の一部を幕全体として扱っていないか
- 幕検査、章検査、帯検査、仮到達点検査を通したか
- LOGを通してから話数仮算出をしたか
- 話数仮算出後に再検査したか
- 結果が動いた場合に未到達扱いしたか
- 結果が動き続ける場合に処理単位を割ったか
- 心理段階を落としていないか
- 生活反復を落としていないか
- 制度処理を便利札にしていないか
- 支援組織を無条件化していないか
- 一回処理の限界に達した場合に成果物風出力を止めたか

到達判定は処理単位ごとに行う。

全体完成と単位到達を混同しない。

## 8.5 POINT間シミュレーション確認

長編、多層作品、判断シリーズ、複数段階の因果を扱う場合に確認する。

確認項目。

- START→CHECKPOINT→GOALだけに固定していない
- START→POINT_01→POINT_02→...→GOALとして扱えるか見た
- POINTを分類棚ではなく因果ノードとして扱った
- POINTに人物、生活、制度、物、禁止、未解決、読後感などの属性をぶら下げた
- POINT_Aへ入る時点の状態を置いた
- POINT_Bで出たい状態を置いた
- POINT_A→POINT_Bで誰が動くかを見た
- POINT_A→POINT_Bで何が足りないかを見た
- 金 / 飯 / 宿 / 制度 / 感情が通るか見た
- 何が閉じ、何が残るか見た
- 禁止解法を使っていないか見た
- 次POINTへ渡せるか確認した
- 詰まったPOINT間だけ設計で補った
- 通った区間だけを粗帯またはV2カードへ起こした

POINT全体を一気に完成させようとしない。

POINT間の小さい帯で通す。

## 12. 1話化確認

1話化できる条件。

- 誰の窓で見ると一番効くかがある
- キャラ反応がある
- 生活制限がある
- 街の代償がある
- 次起点へ渡す条件がある
- 火種がある
- 中盤で段が増える
- 具体物が動く
- 最後に何をどこへ置き直せるかがある
- 次へ残る灯りや痕跡がある

大事件であることは1話化条件ではない。

面白そうであることは1話化条件ではない。

- 置き直しがあるか
- 残響があるか
- 禁止線を守れるか
- 原情報へ戻れるか
- LOGへ戻れるか

- 今の作品状態に合っているか
- 今の執筆さんに渡す形として書きやすいか

## 13.5 V2走行レーン確認

V2カードへ起こす時に確認する。

確認項目。

- V2を詳細全部決めるカードにしていない
- V2を本文が痩せない走行レーン型カードとして扱っている
- 粗帯からV2化できるか判定した
- 今回の核が一文で言える
- START / POINT / GOAL または START / CHECKPOINT / GOAL がある
- 主な人物がどの線で動くか見えている
- 生活圧 / 制度圧 / 感情圧 / 物圧のどれかが働いている
- 終了時に、何が閉じて何が残るか見えている
- 禁止線がある
- 痩せると困る箇所がある
- 執筆さんに任せてよい余白がある
- 深掘りと肉付けを分けた
- 固定しないと因果が壊れるものだけ固定層へ入れた
- 具体は裁量層へ回し、効き目は固定層へ置いた
- 固定層 / 熱量層 / 裁量層を分けた
- 勝手に強めない種がある
- 見ない / 聞かない / 助けない / 支えるがある
- 開示速度がある
- 一時停止と再開がある
- 反復チェックがある
- ラストの残り香がある
- 執筆依頼文を読めば執筆さんが本文へ入れる

V2を生LOGの代替にしない。

V2を深掘りの完全版にしない。

## 14. 薄さ警告

以下は薄い。

- 火種だけある
- 中盤の段がない
- 具体物が飾り
- キャラが反応装置
- 街が背景でしかない
- 痕跡が次に残らない
- 閉じが感想で終わる
- 目的地との接続が弱い
- 次起点への条件渡しがない
- キャラ反応、生活制限、街の代償がない
- 面白そうなイベントだけで立っている
- ユーザー熱源と接続していない
- LOG根拠がない
- V2なのに走行レーンがない
- POINT間の因果がない
- 痩せると困る箇所がない
- 開示速度がない
- 反復差分がない
- 執筆依頼文がない

薄い場合は、厚カード化せず差し戻す。

ユーザーが承知して進める場合は、妥協LOGへ記録する。

## v003ab WORK_UNIT_CARD確認

設計シミュレーション、骨割り、POINT群作成、粗帯作成、V2渡し範囲判定を行う時は、WORK_UNIT_CARDを確認する。

確認項目。

- WORK_UNIT_CARDを作成した
- parent_object がある
- working_range がある
- canonical_anchor がある
- source が設定され、実読済みである
- 090 / 029_TAG / INDEX / END_LOGを作品条件源にしていない
- required_lanes がある
- split_map がある
- shortage_map がある
- 不足なしの場合は「不足なし」と明記している
- simulation_rounds を複数回回した
- convergence を、収束 / 未収束 / 停止で明記した
- split_map が大きく変わらないことを確認した
- shortage_map に新しい重大不足が出ないことを確認した
- 人物配置が反転していない
- canonical_anchorへ戻れる
- 禁止線に触れていない
- V2へ渡す固定層 / 熱量層 / 裁量層が大きく変わらない
- ユーザー裁定が必要な芯変更が出ていない
- local_stop がある
- exclude がある
- output がある
- V2化 / PW化 / 本文化へ滑っていない

旧STOPの `objectが複数`、`canonicalが割れる`、`1作業単位を超える` だけで停止しない。

割れることは正常である。

停止するのは、割れ方、不足、戻り先、収束が見えていない時である。

WORK_UNIT_CARD不成立時のSTOP。

- parent_object が不明
- working_range が不明
- canonical_anchor が不明
- source未設定 / 未読
- required_lanes が不明
- split_map を出さずに細部決定へ入っている
- shortage_map を出さずに通過扱いしている
- simulation_rounds を回さずに1回で確定扱いしている
- convergence 未確認のまま通過扱いしている
- 割れた先が canonical_anchor へ戻れない
- 不足があるのに補完で進めている
- local_stop 未設定
- exclude 未設定で逸脱しそう
- 029_TAG / 090 / INDEX / END_LOGだけで判断している
- V2化 / PW化 / 本文化へ滑っている

WORK_UNIT_CARDがSTOPした場合も、診断、分類、不足一覧、次に必要なsource、ユーザー裁定が必要な芯は返す。
